# Axiom

## Project Overview

Axiom is a Python framework for orchestrating agentic workflows under deterministic, manifest-driven policy control. It is source-first and runs against a local SQLite database using `sqlite-vec` and a local Ollama model (`qwen3:4b`). 

The framework is strictly **local-first, fail-closed, and non-autonomous by design**. It features a 7-step policy evaluation chain and strict manifest binding. No tool runs unless a signed, schema-validated manifest authorizes it.

## Building and Running

There is no entry-point binary. Execution is performed via CLI scripts in the `tools/` directory, `pytest`, or via `python -c "..."`.

**Setup Database and Manifests:**
```powershell
python -m pip install -r requirements.txt
python -c "from axiom.persistence.db import init_db; init_db()"     # Apply schema and enforce PRAGMAs
python tools/register_manifests.py                                  # Validate and register policy artifacts
```

**Verification and Auditing:**
These scripts are read-only and serve as the canonical preflight to check system health:
```powershell
python tools/verify_foundation.py        # foundation_passed + fail-closed coherence
python tools/audit_task_lifecycle.py     # lifecycle audit
python tools/audit_task_execution.py     # execution audit
python tools/audit_policy_security.py    # policy/security audit
pytest tests -v                          # run full regression test suite
```

**Operational Tools (Scheduler / Execution Pipeline):**
- `python tools/bootstrap_check.py` - Passive boot gate (BootstrapValidator)
- `python tools/scheduler_tick.py <session_id>` - One gated tick 
- `python tools/run_scheduler_loop.py <session_id>` - Bounded multi-tick loop

## Development Conventions

- **Coding Style:** 4-space indentation, `from __future__ import annotations`, type hints, explicit absolute imports (e.g., `from axiom.* import ...` - no relative imports).
- **Exceptions:** Use domain-specific exception names (e.g., `PolicyDeniedError`, `BootVerificationError`), typically one per module.
- **Testing:** 
  - Tests mirror the `axiom/` directory layout under `tests/` (e.g., `test_<area>.py`).
  - Cover both allowed and denied paths for policy or state transitions.
  - **CRITICAL:** Tests must NEVER touch the dev DB (`C:\axiom\axiom.db`). Use the autouse `isolate_axiom_db` fixture in `tests/conftest.py` for DB state tests.
- **Manifest Updates:** After editing anything in `axiom/policy/` (manifests, schemas, tool-capability map), you MUST run `python tools/register_manifests.py` then `pytest`.
- **Operational Posture Constraints:** 
  - Do NOT "fix" the non-autonomous state. The system is intentionally fail-closed and non-autonomous (`autonomous_allowed = False`, `safe_pass_enabled = False`).
  - Do NOT enable autonomous operation, safe-pass, model-profile promotion, real cloud model calls, real NetworkGateway fetches, or other runtime authority expansions without explicit authorization.
